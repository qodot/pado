# Pado.LLM 프로젝트 지침

이 문서는 이 저장소에서 작업하는 에이전트·사용자 모두에게 적용된다.

## 언어

- **모든 문서·주석·커밋 메시지는 한국어로 작성한다.**
  - `@moduledoc`, `@doc`, `@typedoc` 포함.
  - 일반 `#` 주석 포함.
  - `README.md`, `AGENTS.md`, `CHANGELOG.md` 등 저장소에 체크인되는 모든 마크다운 문서 포함.
  - 커밋 메시지 subject/body 전부 한국어.
- **내부 추론·작업 메모·도구 사용 전후 판단 문맥도 한국어**를 기본 작업 언어로 사용한다. 에이전트 하네스가 이런 내용을 노출할 수 있으므로, 사용자에게 보이는 답변과 같은 기준으로 한국어 문장으로 작성한다.
- 이 규칙은 용어를 억지로 번역하라는 뜻이 아니다. 코드 식별자, API 필드, HTTP 헤더, 고유명사, 도메인 용어는 원문을 유지하되, 그것을 해석하고 판단하는 문장은 한국어로 쓴다.
- **코드 식별자는 영어**를 유지한다. 모듈·함수·변수·타입·아톰·파라미터·에러 리즌 키워드는 전부 영어.
- **end 사용자에게 전달되는 모든 텍스트는 영어로 쓴다.** 한국어는 오직 문서·주석·docstring 안에서만 허용한다. 예외는 테스트 코드(`test/**`)뿐이다. 테스트 이름·assert 메시지·픽스처 문자열은 한국어를 써도 된다.
  - 적용 대상: `IO.puts`/`IO.write`, `Mix.shell().info/error`, `Logger.{debug,info,warning,error}`, `raise`/`throw` 메시지, 예외체의 `message` 필드, CLI 프롬프트, 사용자 호출자에게 그대로 노출될 것으로 설계된 콜백 메시지/이벤트 메시지, OAuth 콜백 페이지처럼 브라우저·터미널 등으로 그대로 실릴 텍스트 일체.
  - 영어로 쓸 때도 가독성을 자이지 않는다. 완성된 문장으로 쓰고, 항목 포맷이 필요하면 `key=value` 형태나 결과 재쓰기(대문자, 구두점)을 일관되게 맞춘다. 디버깅 메시지도 원인과 맥락이 드러나도록 만들고, 알아볼 수 있는 차원은 `inspect/1`로 붙이는 게 좋다.
  - `inspect/1`로 붙는 구조화된 값(맵·구조체·리스트 등)은 데이터이므로 그대로 둔다.

## 커밋 메시지

- Conventional Commits 스타일 prefix(`feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`)는 유지하되, 뒤의 설명은 한국어로 쓴다.
  - 예: `feat: OpenAI Codex OAuth 로그인 초안 추가`
  - 예: `fix: refresh 토큰 로테이션 시 extra 필드 누락 수정`
- 본문이 필요하면 72자 줄바꿈을 맞추되 내용은 한국어로.

## 커밋 단위

- **한 커밋에 한 가지 맥락만 담는다.** 여러 파일이 바뀌어도 같은 의도면 한 커밋이고, 같은 파일 안이라도 서로 무관한 변경이면 별개 커밋이다.
- 큰 변경은 의도 단위로 쪼개 순서대로 푸시한다. 예:
  1. `feat: Credentials 구조체 추가`
  2. `feat: PKCE 헬퍼 추가`
  3. `feat: OpenAICodex 로그인 플로우 구현`
- **포매팅/리팩터/기능 추가를 섞지 않는다.** 순수 포맷 변경은 `style:` 또는 `chore: format` 단독 커밋.
- WIP·덩어리 커밋은 지양한다. 필요하면 `git add -p`로 헝크 단위 선택.
- 작업 중 깨진 상태를 저장할 때는 별도 브랜치에서만 허용하고, 머지 전에 `git rebase -i`로 정리한다.

## 코드 스타일

- `mix format`을 반드시 통과해야 한다.
- `mix compile --warnings-as-errors`가 통과해야 한다.

## 코드 설계 기준

- 새 필드·함수·모듈을 추가하기 전에 현재 코드에서 실제로 두 개 이상의 사용처가 있는지 확인한다. 사용처가 하나뿐이면 먼저 단순한 이름과 직접 호출을 선택한다.
- 미래 확장을 이유로 `api`, `kind`, `type`, `mode` 같은 구분 필드를 만들지 않는다. 실제로 두 구현이 생긴 뒤에 분리한다.
- 공개 함수는 현재 호출 흐름에 꼭 필요한 것만 둔다. 내부에서 한 줄로 조합할 수 있는 편의 함수는 호출자가 직접 조합하게 둔다.
- 이름은 현재 책임을 그대로 드러내야 한다. 예를 들어 provider 기준 dispatch라면 `adapter_map`보다 `provider_map`을 쓴다.
- 에러 이름은 구현 방식이 아니라 사용자 관점의 실패 이유를 표현한다. 예를 들어 adapter가 없다는 내부 사정보다 `:unsupported_provider`처럼 호출자가 이해할 수 있는 이름을 우선한다.
- 모듈 이름에 엔드포인트나 구현 세부사항을 넣지 않는다. 프로바이더에 호출 방식이 하나뿐이면 `OpenAICodex.Responses`보다 `OpenAICodex`를 쓴다.
- docstring, spec, 주석은 기본적으로 추가하지 않는다. 코드만으로 의도가 드러나지 않거나 외부 사용자가 반드시 알아야 하는 제약이 있을 때만 짧게 쓴다.
- 프로바이더별 응답 모양 차이는 각 프로바이더 내부 매퍼에서 끝낸다. 상위 계층에는 공통 이벤트만 노출한다.
- 스트리밍 API가 핵심이다. 최종 결과만 반환하는 별도 편의 API는 실제 사용처가 생기기 전까지 만들지 않는다.

## OAuth / 크레덴셜 취급

- 라이브러리는 **어떤 영속 저장소도 소유하지 않는다.** 로그인 결과는 호출자에게 `%Pado.LLM.Credential.OAuth.Credentials{}`로 반환만 한다.
- 테스트·스크립트·문서 안에 **실제 토큰을 절대 커밋하지 않는다.** 샘플이 필요하면 명백히 합성된 더미(`"access_xxx"`, `"refresh_yyy"` 등)를 쓴다.
- `refresh/1` 호출 결과는 항상 새 `%Credentials{}`를 돌려주며, 저장소 갱신(로테이션)은 호출자 책임임을 문서에 명시한다.
